我的游戏开发周记（第二周）

发表于 2015-06-04   |   分类于 游戏开发   |  

一点题外话

距离上一篇都过去了两个星期了。完全不是周记的节奏啊。只怪上周有个mapreduce考试耽搁了时间。而且私以为考试成绩与复习时间没有相关性。之所以要说点题外话，是因为上周我又把博客的主题做了点设置。主要修改的有将原有主题的内主题更换了，同时为了美化本地搜索框参考了这篇文章。因为博客主题的开发者还在更新，所以得做好记录，方便以后重配。下面接着上一篇的话题接着往后说。这次的话题可能有些偏学术。

正题

从上一篇博客我们可以知道，第一阶段要做的事情就是初始化相应的变量，并载入所需的图片和音频资源。然后，在资源加载完毕之后，我们就要开始引导整个游戏的开始了。首先得提到一个概念：状态机。为什么需要用到状态机呢？因为一个游戏的运行过程（从开始到结束）其实是一个状态不断改变的过程，在不同的状态所要做的事情也是不同的。简单地说，我们可以把游戏分为载入状态、菜单状态、游戏中状态等等。不同的状态往往对应着不同的绘制画面等等。这里，我们在love.load函数的结尾首先进入一个名为intro的状态，调用intro_load函数。源代码中intro_load()函数非常简单，所以我就把它列出来了。。

从代码中我们可以看到，gamestate设置为intro状态。screenwidth和screenheight是窗口的尺寸，allowskip是设置是否跳过的标志。中间另外有一些其他的参数再后面会解释。那么这个intro状态到底是做什么的呢？给大家3秒钟猜一下。

intro，顾名思义，就是载入的时候发生的动作。什么动作呢？想想看，我们一般在打开游戏的时候，是不一定直接就会出现游戏菜单的对不对？没错，这个intro是用来播放载入/开场/Logo/其他画面的。实际上，在我们的游戏中，就是载入了一个Logo图片，然后播放一段开场音效的。那何时做这些事情？这就是我们接下来要说的。

根据love2d运作的原理，love.load函数只会在程序运行之初加载一次，接下来会一直循环调用love.update和love.draw。两个函数，一个负责逻辑更新，一个负责最终画面的绘制。love.update函数有一个参数dt，代表自上一次调用update以来经过的时间。为了清晰理解整个love2d的运行过程，这里贴一下默认的love.run的实现代码。

我们并不需要完全理解其中的意思。但大体上，可以看出其运行逻辑。首先是三次随机（据说重要的事情连做三次就会效果拔群！但是我忘记了这个三次随机的依据是在哪里看到了的==），其次是初始化消息队列，调用load函数，然后调用timer进行计时，接着进入名为while true的大循环。

大循环的做法符合很多游戏设计逻辑的做法。首先是对消息队列进行处理，如果有退出程序的消息，优先做处理，其他的调用默认的handler函数。处理完消息后，根据计时器得到流逝的timestamp，然后传给update函数调用。最后，再调用draw函数进行最终的绘制并提交。不断重复该过程。

也就是说，下面我们要看的是love.update。不过在说具体怎么做之前，先学习一段关于帧率的理论知识。这个帧率，与update函数的dt参数密切相关。

帧率

好，现在我们开始讲帧率。继我上回写到的地方已经过去了将近一个星期。最近事情有点多呐，基本上每天晚上都要出去各种事。好了，我其实想说的是，得为这份工作保留点时间啊。记得在一篇文章中看到过，有人会每天为自己腾出一个小时来专门做一件事。废话讲完了，现在说正题。

本段落内容参考文章Fix Your Timestep!，再加上自己本身的理解记录而成。请注意：以下的内容与love2d引擎本身无关。

首先，我们要理解两件事，即update和draw。update用于更新物理逻辑，draw用于绘制最终画面。有种很简单的方法，就是让这两者融合在一起，放在一起做。假如我们要以固定的delta时间执行，例如1秒60帧，就可以这样：

double t = 0.0; 
double dt = 1.0 / 60.0;  
while ( !quit ) { 
    integrate( state, t, dt ); 
    render( state ); 
    t += dt; 
}

理想情况下integrate和render可以协调工作。然后问题就来了：我们压根并不知道显示的刷新频率。实际上刷新频率也不是固定的（在垂直同步关闭的情况下），它有可能非常快，1秒不止60次，也有可能非常慢，远远跟不上逻辑的更新速率。oh-no！你的世界已经出问题了。

显然，我们希望的是，逻辑的更新频率随着显示的刷新频率变化而变化。这时候，就需要让delta时间变动了，它不能固定为1/60秒。

double t = 0.0;  
double currentTime = hires_time_in_seconds();  
while ( !quit ) { 
    double newTime = hires_time_in_seconds(); 
    double frameTime = newTime - currentTime;
    currentTime = newTime;  
    integrate( state, t, frameTime ); 
    t += frameTime;  
    render( state ); 
}

有没有觉得这段代码如此得眼熟呢。是的，它很像love.run的写法。不过，我们在这里主要考察的是逻辑部分。这里我们要假定一个事实：就是integrate的逻辑更新与delta时间有关。举个栗子：每次调用integrate函数，我们就让主人公前进1*delta个像素点。但是这样是有一点问题的，问题在什么地方呢？如果delta时间过长，远超过你预先设计的最大值（比如1/60秒），主人公就会在一次逻辑更新中前进非常大的值，导致不可预料的后果（比如撞墙什么的）。所以，接下来我们要对这种极端情况进行限制。

所谓进行限制，就是让这种情况永远不会出现，即我们要用一些小trick来确保不会传递比设计的最大值还要大的值到integrate里。

double t = 0.0; 
double dt = 1 / 60.0;  
double currentTime = hires_time_in_seconds();  
while ( !quit ) { 
    double newTime = hires_time_in_seconds(); 
    double frameTime = newTime - currentTime; 
    currentTime = newTime;  
    while ( frameTime > 0.0 ) { 
        float deltaTime = min( frameTime, dt ); 
        integrate( state, t, deltaTime ); 
        frameTime -= deltaTime; 
        t += deltaTime; 
    }  
    render( state ); 
}

我们可以看到，经过这样的处理后，deltaTime拥有一个上界。如果更新时间过长，我们将frameTime分为多个deltaTime的片段，这样就可以逐步更新了。当然，如果更新时间很短，也不会有什么影响。

然而，这样做却还是存在着一些问题。可以发现，当frameTime很大时，我们需要多次调用integrate进行逻辑更新。这时，有可能这多次的调用（尤其是最后一次，因为最后一次的deltaTime可能非常非常小）成为了整个过程中最昂贵的开销。这会导致什么？会导致实际开销的时间超过了预留的时间。也就是说，虽然逻辑更新了，但是它依旧落在真实的物理时间后面。通俗地说，就是一次逻辑更新花费的时间要比frameTime多，这会导致下一次的frameTime变大。如此往复，就好像滚雪球一样，你需要调用更多次的integrate，而frameTime也会相应变得更大，形成恶性循环。

所以呐，凡事都要留白。为了避免上述悲剧的发生，我们需要保证更新X秒的物理逻辑所花费的时间一定要小于X秒。这样做的话，偶尔会遇到某个时刻需要更新更多次的物理逻辑，不过也可以设置一个每个渲染帧最多可调用更新物理逻辑的次数作为upper bound。不管怎么说，总比滚雪球的恶性循环要好。

当然，如果使用固定的delta时间作为物理逻辑更新，效果会精确地多。如果能在不同的帧率下还能保持更新的delta不变，那就更好了。那该怎么去做呢？这时我们可以转换一下视角为生产者消费者问题。渲染是产生时间的，逻辑更新是消耗时间的，并且它每次消耗的是离散的固定时间块。

double t = 0.0; 
const double dt = 0.01;  
double currentTime = hires_time_in_seconds(); 
double accumulator = 0.0;  
while ( !quit ) { 
    double newTime = hires_time_in_seconds(); 
    double frameTime = newTime - currentTime;
    currentTime = newTime;  
    accumulator += frameTime;  
    while ( accumulator >= dt ) { 
        integrate( state, t, dt ); 
        accumulator -= dt; 
        t += dt; 
    }  
    render( state ); 
}

不过，这里留下的不足dt的时间可能成为一种隐患。想象一下帧率为60fps而我们期望的物理更新为50fps的情形。这种情况下，每隔几次物理就连续更新两次来追上帧渲染。由于时常有物理落后于帧渲染的情形，这样会导致屏幕上的画面渲染出现卡顿，被称之为“时间走样（temporal aliasing）”的问题。

解决这个问题的方法，就是利用插值，在先前与当前物理状态之间插入新的状态。

double t = 0.0; 
double dt = 0.01;  
double currentTime = hires_time_in_seconds(); 
double accumulator = 0.0;  
State previous; 
State current;  
while ( !quit ) { 
    double newTime = time(); 
    double frameTime = newTime - currentTime; 
    if ( frameTime > 0.25 ) 
        frameTime = 0.25; 
    currentTime = newTime;  
    accumulator += frameTime;  
    while ( accumulator >= dt ) { 
        previousState = currentState; 
        integrate( currentState, t, dt ); 
        t += dt; 
        accumulator -= dt; 
    }  
    const double alpha = accumulator / dt;  
    State state = currentState * alpha +  previousState * ( 1.0 - alpha );  
    render( state ); 
}

最后accmulator的含义是物理的更新进度。比如，accmulator最后值为dt/2，这就意味着物理已经更新到一半了，还有一半就可以更新到下一帧了。因此当前需要渲染的画面应当是一个插值的结果。上面代码用的是线性插值。不过有两点值得注意：

1、代码中使用的是内插。换言之，插值得到的是当前状态和先前状态的插值，也就是要比真实情况要晚一些。为什么不预先计算出下一状态，然后用当前状态进行外插处理呢？作者在评论中给出的解释是：

the reason i interpolate previous two frames is to always be accurate, you could extrapolate from the current frame if you want, but this causes discontinuities if the velocity changes abruptly (eg. a bounce…)

If you interpolate you’ll see no penetration, but on bounces you’ll see the curve flatten out strangely. consider a bounce like a “V” but the bottom point of the V is not a sample point, now the bounce is flattened out.

If you interpolate instead of extrapolate, you are “guessing” ahead, and in the V case you’ll actually guess a bit into the surface before bouncing out, at this point you’ll see discontinuity due to extrapolation. So don’t extrapolate.

2、代码中有一句if ( frameTime > 0.25 ) 。这句话的作用是如果画面渲染已经慢到连1秒4帧都渲染不了，就假设它1秒能渲染4帧吧！如果真的卡了，就让画面卡着吧。。。

好了，本次的内容就到这里了。关于帧率，最简单的写法当然是不固定了。在不同的场合用不同的写法吧。
